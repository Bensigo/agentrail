# AgentRail Docs Site (`/docs`) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a Fumadocs-powered documentation site at `/docs` inside the console app that teaches developers how to set up and use the AgentRail CLI, then wire it into the marketing nav/footer.

**Architecture:** Mount Fumadocs inside `apps/console` as a `(docs)` route group with its own `RootProvider` + `DocsLayout`, so its theme/sidebar/search are scoped to `/docs` and the marketing/dashboard layouts are untouched. MDX content lives in `apps/console/content/docs/`, compiled by `fumadocs-mdx` via the Next plugin. Content is derived from the real CLI command surface (`agentrail/cli/main.py`), not the drifted README.

**Tech Stack:** Next.js 15.3 (App Router), React 19, Tailwind v4, `fumadocs-ui` + `fumadocs-core` + `fumadocs-mdx`, pnpm monorepo.

**Source of truth for spec:** `docs/superpowers/specs/2026-06-19-docs-site-design.md`

**Branch:** Create `feat/docs-site` off `origin/main` before starting. Do NOT build on `feat/landing-redesign` (that's PR #861). The working tree currently has unrelated WIP (context-quality, db-clickhouse) — leave it untouched; stage only files this plan names.

---

## Verification model

This is config + UI + content work, so the "tests" are build/typecheck/route/browser gates, not vitest units (there is no meaningful pure unit to TDD for a docs renderer). Where a pure helper exists (none required here), a unit test would be added. Each task ends with a concrete verification command and expected output. Browser verification uses the preview tools per the harness workflow.

The dev server runs via `pnpm dev` (port 3000); logs at `/tmp/agentrail-dev.log`. Restart it after dependency installs and config changes.

### Task 6: Docs layout (scoped RootProvider)

**Files:**
- Create: `apps/console/app/(docs)/layout.tsx`

- [ ] **Step 1: Create the layout**

`theme={{ enabled: false }}` prevents Fumadocs' next-themes provider from fighting the app's own dark-mode script in the root layout (which forces `.dark` on `<html>`).

```tsx
import { RootProvider } from "fumadocs-ui/provider/next";
import { DocsLayout } from "fumadocs-ui/layouts/docs";
import { baseOptions } from "@/lib/layout.shared";
import { source } from "@/lib/source";
import type { ReactNode } from "react";

export default function Layout({ children }: { children: ReactNode }) {
  return (
    <RootProvider theme={{ enabled: false }}>
      <DocsLayout {...baseOptions()} tree={source.getPageTree()}>
        {children}
      </DocsLayout>
    </RootProvider>
  );
}
```

Note: `@/lib/...` resolves via the existing `@/*` → `./app/*` alias.

### Task 7: Catch-all docs page

**Files:**
- Create: `apps/console/app/(docs)/docs/[[...slug]]/page.tsx`

- [ ] **Step 1: Create the page**

```tsx
import { source } from "@/lib/source";
import {
  DocsBody,
  DocsDescription,
  DocsPage,
  DocsTitle,
} from "fumadocs-ui/page";
import { notFound } from "next/navigation";
import { getMDXComponents } from "@/components/mdx";
import type { Metadata } from "next";

export default async function Page(props: {
  params: Promise<{ slug?: string[] }>;
}) {
  const params = await props.params;
  const page = source.getPage(params.slug);
  if (!page) notFound();

  const MDX = page.data.body;

  return (
    <DocsPage toc={page.data.toc}>
      <DocsTitle>{page.data.title}</DocsTitle>
      <DocsDescription>{page.data.description}</DocsDescription>
      <DocsBody>
        <MDX components={getMDXComponents()} />
      </DocsBody>
    </DocsPage>
  );
}

export async function generateStaticParams() {
  return source.generateParams();
}

export async function generateMetadata(props: {
  params: Promise<{ slug?: string[] }>;
}): Promise<Metadata> {
  const params = await props.params;
  const page = source.getPage(params.slug);
  if (!page) notFound();
  return {
    title: page.data.title,
    description: page.data.description,
  };
}
```

Note: import paths `fumadocs-ui/page` and `fumadocs-ui/layouts/docs` are the v15 layout entrypoints. If typecheck reports them missing, check `node_modules/fumadocs-ui/package.json` `exports` for the exact subpaths (older docs used `fumadocs-ui/page`; newer use `fumadocs-ui/layouts/docs/page`) and adjust.

```mdx
---
title: Introduction
description: What AgentRail is and who it's for.
---

AgentRail is the control plane that runs your coding agents. You bring an agent
— Claude Code, Codex, or Cursor — and AgentRail gives it durable repo context,
bounded execution, independent review, and verification gates, so its work is
cheaper, inspectable, and safe to leave unattended.

AgentRail is **not** a coding agent. It does not write code itself. It runs the
agent you already use, on rails: scoped context instead of whole files, a hard
budget per issue, a second model reviewing the work, and gates that stop merges
the agent can't justify.

## Who it's for

- **Engineers** who want to hand an issue to an agent and get a reviewed PR back
  without babysitting a run or watching the token meter.
- **Startups** clearing a backlog overnight under a predictable spend ceiling.

## Two ways to run it

- **Repo-native (primary):** install the CLI, point it at your repo, and run
  issues locally with your own LLM keys. Covered in
  [Installation](/docs/getting-started/installation) and
  [Quickstart](/docs/getting-started/quickstart).
- **Hosted runner (team layer):** sign in to a workspace, queue issues in the
  console, and run a local worker that pulls and executes them. Covered in
  [Dashboard & hosted runner](/docs/dashboard).
```
