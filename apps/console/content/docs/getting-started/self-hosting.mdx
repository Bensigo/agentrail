---
title: Self-hosting
description: Run the worker yourself, or the whole stack — your machine, your LLM keys, your custody of the code.
---

Hosted execution is the default: message Jace and the work runs in the cloud
(see the [Quickstart](/docs/getting-started/quickstart)). This page covers
running things yourself instead, in two flavors: attach your own worker to
the hosted console (steps 1–6 below), or run the entire stack on
infrastructure you control (console, Jace, database, and worker).

## 1. Sign up

Go to [heyjace.com](https://heyjace.com) and create an account (or sign in if
you already have one).

## 2. Connect GitHub

On the **Connectors** page, authorise GitHub and select the repos you want
AgentRail to watch. Set the trigger label to `ready-for-agent` and choose a
poll interval. AgentRail will enqueue any issue you label `ready-for-agent`.

## 3. Install the CLI

```bash
npm install -g @useagentrail/cli
```

This installs the `agentrail` command (also available as `useagentrail`).
Verify your install:

```bash
agentrail doctor
```

## 4. Log in

```bash
agentrail login
```

`login` runs an OAuth device flow that links this machine to your workspace
and saves credentials to `~/.agentrail/credentials.json`. Login is required —
the only exempt commands are `login`, `logout`, `whoami`, and `--help`.

## 5. Start the runner

```bash
agentrail runner
```

The runner is the local worker that polls your workspace for queued issues,
executes them on your machine with your own LLM keys, opens PRs, and reports
outcomes back to the dashboard. Keep it running in a terminal (or as a
background service) while you work.

```bash
agentrail runner --concurrency 2   # run up to 2 issues in parallel
```

## 6. Label an issue — get a reviewed PR

On any connected GitHub repo, add the `ready-for-agent` label to an open
issue. The dashboard enqueues it, your runner picks it up, runs a bounded
plan → execute → verify loop, and opens a PR. Review the PR and merge.

## Running the whole stack yourself

Steps 1–6 attach your own runner to heyjace.com's console — Jace and the
dashboard stay hosted. To run everything yourself instead — console,
database, Jace, and the worker, all on infrastructure you control — follow
the single-VM deploy walkthrough:
[`deploy/README.md`](https://github.com/Bensigo/agentrail/blob/main/deploy/README.md).

Two things become yours to configure once Jace itself is self-hosted:

### Telegram: bring your own bot

The hosted product runs one shared Telegram bot for every workspace. Running
your own `apps/jace` instead, you register your own
[BotFather](https://t.me/BotFather) bot: message @BotFather → `/newbot` to
get a token, then message your new bot once before connecting it. Point
Telegram's `setWebhook` at your own deployment's `/eve/v1/telegram` — see
[`apps/jace/README.md`'s Channels section](https://github.com/Bensigo/agentrail/blob/main/apps/jace/README.md#channels)
for the exact call, and the same section for Discord, Slack, and iMessage
setup.

### Jace's model: bring your own

By default Jace's coordinator runs on Claude via the Vercel AI Gateway.
Self-hosting `apps/jace`, you can point it at any OpenAI-compatible endpoint
instead — a local Ollama, vLLM, LM Studio, or LiteLLM — by setting
`JACE_MODEL_BASE_URL`, with `JACE_MODEL_ID` selecting the model and
`JACE_MODEL_API_KEY` an optional bearer token. See
[`apps/jace/docs/HOSTING.md`](https://github.com/Bensigo/agentrail/blob/main/apps/jace/docs/HOSTING.md#authentication)
for the full picture, including the context-window override a self-hosted
model needs. This is separate from the runner's own coding-agent keys
(Claude, Codex, or Cursor) — see [Configuration](/docs/configuration) for
those.
